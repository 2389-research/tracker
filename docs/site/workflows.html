Key Design Decisions

Split Escalation Gates

EscalateMilestone (mid-build): mark done, retry, accept, abandon. Defaults to "mark done" so you can override flaky tests without aborting the build.

EscalateReview (post-build): accept, retry, abandon. Defaults to "accept" for shipping.

Tests Run Before Counter

TestMilestone runs the test suite before checking the attempt counter. If a milestone was genuinely fixed on attempt 4, it succeeds instead of escalating.

Per-Milestone Circuit Breaker

A fix_attempts file on disk limits retries per milestone to 3. Persists across pipeline restarts. Reset by MarkMilestoneDone.

Three-Model Cross-Review

Claude, Codex, and Gemini independently review the complete implementation. Reviews are synthesized to find consensus vs disputed findings.

Defensive Shell Scripts

Tool nodes never eval LLM-generated content. Milestone headers are matched flexibly (handles ##, ###, with or without colons). Empty-file guards fail loudly. Known-failures strips comments before building patterns.

Autopilot Compatible

Run with --autopilot mid for fully autonomous execution. All human gates (ApprovePlan, EscalateMilestone, EscalateReview) have default edges and descriptive prompts that the LLM judge can reason about.

# Run it
tracker build_product

# Copy and customize
tracker init build_product
# edit build_product.dip, then:
tracker build_product.dip
Competitive Fan-Out

ask_and_execute

Ask the user what to build, explore and spec it, fan out to 3 isolated implementations via git worktrees, cross-critique, select the best, apply it, clean up the rest.

How It Works

  1. Understand

    Human describes the task. An exploration agent maps the codebase. A spec agent produces a detailed implementation plan.

  2. Implement in Isolation

    Three agents (Claude, OpenAI, Gemini) each implement the spec in their own git worktree. Full isolation, no interference.

  3. Cross-Critique & Select

    Each implementation is reviewed by all three models. A judge selects the best one based on aggregate scores.

  4. Apply & Clean

    The winning implementation is merged to main. Losing worktrees are deleted.

Designed For

Large Specs

Multi-stream specs with explicit dependency graphs between work streams.

Mechanical Quality Gates

Build + test + lint + coverage + complexity checked at each phase boundary. Not LLM-judged.

Traceability Audit

Final gate verifies every spec requirement maps to implementation and test coverage.

Backend Agnostic v0.12.0

All three built-in workflows work with both the native and claude-code backends. No changes needed — just add --backend claude-code to give agents file editing, terminal access, and MCP tool servers. Mix backends per-node by adding backend: claude-code to individual agent definitions in your customized .dip file. 

# Same pipeline, supercharged agents
tracker --backend claude-code build_product

# Or combine with autopilot for fully autonomous + file editing
tracker --backend claude-code --autopilot mid build_product

Pipeline testing

Every edge from every gate is tested with dippin simulation tests. Scenario overrides inject specific conditions; assertions verify the exact routing path. 

{
  "name": "milestone escalation — mark done continues build",
  "scenario": {
    "TestMilestone.tool_stdout": "escalate",
    "TestMilestone.outcome": "fail"
  },
  "expect": {
    "immediately_after": {"EscalateMilestone": "MarkMilestoneDone"}
  }
}

dippin test examples/build_product.dip
