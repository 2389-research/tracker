Architecture

Three layers, clean boundaries, no magic. The engine doesn't know about parallel execution. The LLM client doesn't know about pipelines.

Handler types

Every node resolves to a handler. The engine calls Execute() and reads the outcome; it has no special-case code for any handler type. New handlers can be registered without touching the run loop.

HandlerNode kindWhat it does

tooltoolShell command execution with per-node timeout, output limits, env scoping, denylist/allowlist, and comment/blank stripping for LLM-written lists.
humanhumanChoice, freeform, hybrid, yes/no, and interview gates. Pluggable via Interviewer for autopilot, auto-approve, or webhook routing.
parallelparallelDispatches branches from parallel_targets in goroutines. Honors max_concurrency and branch_timeout. Points the engine at a fan-in via suggested_next_nodes.

wait.human—Pause for a human signal without presenting a gate UI. Typical pattern for external notifiers that unblock via a callback.

ensureStartExitNodes only converts bare codergen nodes with no prompt into passthrough start/exit nodes. Every other handler type is preserved, so tool, human, parallel, conditional, subgraph, stack.manager_loop, and wait.human nodes can all serve as start or exit.

The engine run loop

The engine treats every node the same: execute the handler, evaluate outgoing edges, advance to the next node. There is no special-case code for parallel execution, subgraphs, or human gates — handlers encapsulate all node-type-specific behavior.

Edge Selection Priority

When a node completes, the engine selects the next edge in this order:

  1. Suggested Next Nodes

    If the handler set suggested_next_nodes (e.g., the parallel handler pointing to a fan-in), take that edge. Used for structural routing.

  2. Conditional Match

    Evaluate conditions in declaration order. First match wins. Supports =, !=, contains, not contains, startswith, endswith, in, and logical and/or/not.

  3. Label Match

    If the outcome includes a preferred_label (from a human gate), match it against edge labels.

  4. Default Edge

    First unconditional edge. This is the safety net — and the source of infinite loops if pointed at a loop target.

Infra-Level Retry

Handles transient provider errors (502, 503, 429, timeouts) transparently. Max 3 retries, 2s base delay with exponential backoff. Pipeline-level retries are reserved for actual node logic failures.

SSE Error Handling

OpenAI's Responses API returns HTTP 200 and sends error / response.failed as SSE events. The adapter parses these — they are NOT reflected in the HTTP status code.

TUI model

Built with bubbletea. The pipeline runs in a background goroutine; the TUI receives events via tea.Msg.

Pipeline Panel

Node list in topological order (Kahn's algorithm). Status lamps, thinking spinners, tool execution indicators. Subgraph nodes dynamically inserted and indented.

Activity Log

Append-only, line-level styled. No glamour rendering (too expensive for streaming). Per-node streams with separators on node change. Counts terminal rows, not entries.

Modal Overlays

Human gates open modal overlays: choice radio, freeform textarea, hybrid radio+freeform, or fullscreen review with glamour viewport. All implement Cancellable.

Event Flow

Engine emits pipeline events → adapted to tea.Msg → sent to program → state store updates → view re-renders. No polling.

Agent backends

AgentBackend in pipeline/backend.go is a minimal contract — one method that executes a node and returns an agent.Event stream. The engine doesn't know which backend is running; it just routes edges based on the outcome.

Per-Node Priority

A node with backend: native always uses native even under --backend claude-code. Priority: per-node attr > global flag > default native. Enables mixed-backend pipelines — implementation on Claude Code, cross-review on OpenAI native.

Native Backend

Wraps agent.Session — the canonical turn loop with tool registry, context compaction, optional plan-before-execute, episodic memory across retries, and structured reflection on tool failure. Works with Anthropic, OpenAI, Gemini, and OpenAI-Compat.

Claude Code Backend v0.12.0

Spawns claude as a subprocess and parses NDJSON. Gets file editing, terminal execution, and MCP tool servers for free. API keys stripped by default so the CLI uses subscription auth (Max/Pro OAuth). Exit codes map to pipeline outcomes.

ACP Backend v0.16.0

Spawns any agent speaking the Agent Client Protocol over stdio. Provider-based default routing (anthropic → claude-agent-acp, openai → codex-acp, gemini → gemini --acp); override via acp_agent node attr. Full terminal and file-ops RPC surface.

Design Note

Lazy LLM client: buildLLMClient() failure is non-fatal when the global --backend is not native. The native client is only required for nodes that actually need it, so you can run a pipeline with no API keys at all if every node is on claude-code (subscription auth) or acp (bridge native auth).

Context & steering

The PipelineContext is the typed key-value store shared across nodes. Variable expansion resolves ${ctx.*}, ${params.*}, ${graph.*}, and ${ctx.node.<id>.*} in prompts, tool commands, and edge conditions — single-pass, never re-scanning resolved values.

Per-node scoping v0.17.0

After each node runs, every key it wrote is also copied into node.<nodeID>.<key>. Downstream nodes read ${ctx.node.ReadSpec.last_response} to reference a specific upstream output instead of the latest one. Bare keys keep last-writer-wins global semantics.

Declarative writes: / reads: v0.21.0

Nodes can declare the keys they produce and consume. Declared writes are validated into the context after the handler runs — missing required fields fail the node. reads: pins fidelity for keys a node consumes so downstream data is consistent.

Steering channel v0.20.0

pipeline.WithSteeringChan(chan map[string]string) lets a supervisor push updates between node executions. MergeWithoutDirty writes into the global namespace without leaking into any node's per-node scope. Used by stack.manager_loop to steer child pipelines mid-run.

Budget governance v0.17.0

The BudgetGuard evaluates between every pair of nodes. A trip on max_total_tokens, max_cost_cents, or max_wall_time halts the run with OutcomeBudgetExceeded and fires EventBudgetExceeded with a full CostSnapshot. Limits come from Config.Budget, CLI flags, or workflow defaults:.

Checkpoint resume

Every node completion saves a checkpoint with completed nodes, context snapshot, and edge selections. Resume replays the checkpoint and picks up at the failure point.

Known Limitation

The restart counter is global across the entire run. A fix loop on milestone 1 consumes restart budget that milestone 10 needs. The build_product pipeline works around this with per-milestone circuit breakers (fix_attempts file on disk).
