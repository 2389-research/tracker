Key Design Decisions

Split Escalation Gates

EscalateMilestone (mid-build): mark done, retry, accept, abandon. Defaults to "mark done" so you can override flaky tests without aborting the build.

EscalateReview (post-build): accept, retry, abandon. Defaults to "accept" for shipping.

Tests Run Before Counter

TestMilestone runs the test suite before checking the attempt counter. If a milestone was genuinely fixed on attempt 4, it succeeds instead of escalating.

Per-Milestone Circuit Breaker

A fix_attempts file on disk limits retries per milestone to 3. Persists across pipeline restarts. Reset by MarkMilestoneDone.

Three-Model Cross-Review

Claude, Codex, and Gemini independently review the complete implementation. Reviews are synthesized to find consensus vs disputed findings.

Defensive Shell Scripts

Tool nodes never eval LLM-generated content. Milestone headers are matched flexibly (handles ##, ###, with or without colons). Empty-file guards fail loudly. Known-failures strips comments before building patterns.

Autopilot Compatible

Run with --autopilot mid for fully autonomous execution. All human gates (ApprovePlan, EscalateMilestone, EscalateReview) have default edges and descriptive prompts that the LLM judge can reason about.

# Run it
tracker build_product

# Copy and customize
tracker init build_product
# edit build_product.dip, then:
tracker build_product.dip
Competitive Fan-Out

ask_and_execute

Ask the user what to build, explore and spec it, fan out to 3 isolated implementations via git worktrees, cross-critique, select the best, apply it, clean up the rest.

How It Works

  1. Understand

    Human describes the task. An exploration agent maps the codebase. A spec agent produces a detailed implementation plan.

  2. Implement in Isolation

    Three agents (Claude, OpenAI, Gemini) each implement the spec in their own git worktree. Full isolation, no interference.

  3. Cross-Critique & Select

    Each implementation is reviewed by all three models. A judge selects the best one based on aggregate scores.

  4. Apply & Clean

    The winning implementation is merged to main. Losing worktrees are deleted.

Three Interview Gates

Scope Interview

After codebase exploration, structured questions narrow the review to what matters: primary concern, target area, deployment context, known incidents.

Findings Interview

After parallel analysis, each finding is presented as a question with options: intentional, known bug, or needs investigation. Your context eliminates false positives.

Priority Interview

Confirmed issues get prioritization questions: timeline, fix approach, dependencies, who reviews. Drives the remediation plan structure.

Interview Mode UX

Each interview gate presents a fullscreen form parsed from the agent's markdown questions. Questions with inline options like Auth model? (API key, OAuth, JWT) become select fields. Yes/no questions become toggles. Open-ended questions become text areas. Navigate with arrow keys, Tab for elaboration, Ctrl+S to submit, PgUp/PgDn for pagination. Previous answers pre-fill on retry. 

# Review any codebase
tracker deep_review

# Fully autonomous review (LLM answers the interviews)
tracker --autopilot mid deep_review

# Copy and customize the review pipeline
tracker init deep_review

Backend Agnostic

All four built-in workflows work with every agent backend: native, claude-code, and acp. Flip globally with --backend, or mix per-node by adding backend: to individual agent definitions in your customized .dip file. Per-node attrs always win over the global flag. 

# Native (default) — pick your own model via provider attr
tracker build_product

# Claude Code — file editing, terminal, MCP servers, subscription auth
tracker --backend claude-code build_product

# ACP — bridge to claude-agent-acp / codex-acp / gemini --acp
tracker --backend acp build_product

Run them unattended

All four workflows are designed for hands-on runs and for fully unattended operation. Pick the gate-handling strategy that matches your deployment.

Autopilot Personas

An LLM judge answers every human gate. Four personas: lax (forward progress), mid (balanced, default), hard (high bar), mentor (approve with detailed feedback).

tracker --autopilot mid build_product

Auto-Approve

Deterministic approval — no LLM call, always picks the default or first option. Best for CI and pipeline-flow tests.

tracker --auto-approve build_product

Webhook Gates

Every gate is POSTed as JSON to your URL; the pipeline blocks until a callback arrives. Slack bots, email approval, factory workers, anywhere. Per-gate tokens, configurable timeout. v0.17.0

tracker --webhook-url https://... build_product

Workflow customization

After tracker init <name>, the local .dip file takes precedence on subsequent runs. Tweak per-node models, backends, or prompts without forking the whole pipeline.

Workflow Params

Top-level vars become ${params.*} graph attrs referenceable in prompts, commands, and conditions. Override at runtime with repeatable --param key=value. Unknown keys hard-fail at startup. v0.19.0

Budget Ceilings

Put max_total_tokens / max_cost_cents / max_wall_time in a workflow's defaults: block; CLI flags still override per-run. Halts emit EventBudgetExceeded. v0.19.0

Human-Gate Timeouts

Set timeout and timeout_action per human gate so unattended runs keep moving if no one responds. Works with freeform, choice, and interview modes. v0.19.0

Declarative writes: / reads:

Declare the context keys a node produces or consumes. Declared writes are validated after the node runs; reads: pins fidelity for consumed keys. Replaces node-type-specific workarounds for typed output. v0.21.0

Manager loops in .dip

Declare stack.manager_loop supervisors directly in .dip via ir.NodeManagerLoop. Six flat attrs: subgraph_ref, poll_interval, max_cycles, stop_condition, steer_condition, steer_context. No more DOT escape hatch. v0.23.0

Manager loop — native .dip syntax v0.23.0

Declare a supervisor that launches a child pipeline, polls it, and optionally steers it mid-execution. Unprefixed attr names are the v0.22.0 contract; the handler also accepts the legacy manager.* names for backward compatibility (unprefixed wins when both are set). steer_context uses canonical sorted k=v,k=v pairs with percent-encoding for ,, =, and %. 

manager_loop Supervise
  label: "Watch build attempts"
  subgraph_ref: "subgraphs/build.dip"
  poll_interval: 30s
  max_cycles: 10
  stop_condition: "stack.child.status = success"
  steer_condition: "stack.child.cycles > 3"
  steer_context: "hint=try-alternate-plan,priority=high"

Pipeline testing
